## 一、项目概览

| 项目 | 语言 | Stars | 定位 | 许可证 |

### 2.2 Cookie / 登录需求

| 项目 | Cookie 需求 | Cookie 管理方式 |
|------|-------------|-----------------|

### 2.3 签名 / 反风控

| 项目 | 签名方案 | 反风控措施 |

| **DouK-DL** | a_bogus | 代理支持、多线程、增量下载、max_pages 控制 |
| **lux** | X-Bogus (goja JS Runtime) | 自动 Cookie 生成（msToken + ttwid） |
| **yt-dlp** | Mobile API (不需要 Web 签名) | Device ID 随机、odin_tt 随机 |

## 五、能力差距分析

### 5.1 VSD 已有的独特优势 (护城河)

| 优势 | 说明 |
|------|------|
| **订阅式自动下载** | 唯一内置 scheduler 的项目，定时检查 + 增量下载 |
| **NFO 刮削** | 唯一生成 Emby/Jellyfin 兼容 NFO 的项目 |
| **多平台统一** | 抖音 + B站 + YouTube 在同一工具中 |
| **Go 单二进制** | 部署简单、内存低、启动快 |
| **Docker 开箱即用** | docker-compose 一键启动 |
| **Web UI** | 现代化 SPA，订阅/队列/历史管理 |

### 5.2 VSD 欠缺的能力 (按价值排序)

| 优先级 | 能力 | 竞品参考 | 价值 | 实现难度 |
|--------|------|---------|------|---------|
| **P0** | a_bogus 签名 | f2 (已开源满血版) | 决定抖音 API 成功率，当前仅 X-Bogus 已被风控 | 中（可参考 f2 开源的 ab 算法，嵌入 goja） |

## 六、建议行动优先级

### 近期 (1-2 周) — 补齐抖音核心短板

1. **a_bogus 签名实现** — 参考 f2 已开源的满血版 ab 算法，移植到 Go/goja。这是最影响用户体验的瓶颈，X-Bogus 已逐渐被风控拦截。
2. **抖音合集下载** — API 端点已在 `endpoints.go` 定义 (`MixVideosAPI`)，实现 `GetMixVideos` + 调度器集成即可。
3. **抖音喜欢列表** — API 端点已在 `endpoints.go` 定义 (`UserLikedAPI`)，类似用户视频列表的翻页逻辑。

### 中期 (1-2 月) — 扩展平台 & 场景

4. **TikTok 支持** — 可参考 yt-dlp 的 Mobile API 方案（不需要 Web 签名），复用现有的下载/调度框架。
5. **抖音直播录制** — 获取拉流地址 + ffmpeg 录制，集成到订阅调度中（"开播自动录制"）。
6. **收藏/收藏夹下载** — 需要登录态 Cookie，API 类似。

### 长期 (3 月+) — 锦上添花

7. **Cookie 池管理** — 多账号轮换，Web UI 管理界面。
8. **评论数据采集** — 可选导出为 JSON/CSV。
9. **搜索 + 发现** — 在 Web UI 中搜索抖音内容并一键订阅。

## 七、结论

**VSD 的核心竞争力是 "订阅式自动下载 + NFO 刮削 + 媒体服务器兼容"** —— 这个组合在开源社区中是独一无二的。

竞品 f2 和 DouK-DL 在抖音 API 覆盖面上更全面（尤其是 a_bogus 签名、合集、收藏等），但它们都缺乏：
- 内置的定时订阅调度器
- NFO 元数据生成
- Emby/Jellyfin 集成能力
- 单二进制 Docker 部署

**最紧迫的短板是 a_bogus 签名**——f2 已开源满血版算法，我们应优先移植。其次是合集和喜欢列表支持，这两个 API 端点已在代码中定义，实现成本低。

VSD 不需要成为 "最全面" 的抖音工具——它的价值在于 **自动化 + 整理 + 媒体服务器兼容** 这个闭环。只需补齐签名和核心内容类型，就能在差异化赛道上保持领先。
