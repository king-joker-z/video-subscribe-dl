# Phase 1.3: Performance & Resilience - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

两处外科手术式修复：

1. **Dashboard 统计单次查询**：`GetStatsDetailed()` 当前发出 7 个独立 `QueryRow` 调用；改为 2 个查询（1 个 downloads 聚合 + 1 个 sources 计数），`HandleDashboard` 的 DB 查询数从 8 降到 2。
2. **abogus 池超时**：`abogusPool.sign()` 当前对 `ap.pool` 做裸 channel receive，池耗尽时会永久阻塞；镜像 `signPool.sign()` 已有的 `context.WithTimeout` + `select` 模式，加 10s 超时。

不包含：新平台支持、指标暴露、前端变更、其他 Douyin 方法的 ctx 改造。

</domain>

<decisions>
## Implementation Decisions

### GetStatsDetailed SQL 结构
- **D-01:** 使用 **2 个查询**，不用 JOIN 合并到 1 个
  - 查询 1：针对 `downloads` 表，一条 `SELECT` 取出全部聚合字段（`COUNT(*) AS total`、`SUM(CASE WHEN status IN ('completed','relocated') THEN 1 ELSE 0 END) AS completed`、`SUM(CASE WHEN status IN ('failed','permanent_failed') THEN 1 ELSE 0 END) AS failed`、`SUM(CASE WHEN status='pending' THEN 1 ELSE 0 END) AS pending`、`SUM(CASE WHEN status='charge_blocked' THEN 1 ELSE 0 END) AS charge_blocked`、`COALESCE(SUM(CASE WHEN status IN ('completed','relocated') THEN file_size ELSE 0 END),0) AS total_size`）
  - 查询 2：`SELECT COUNT(*) FROM sources WHERE enabled = 1`
  - 理由：sources 计数与下载状态无关联关系，强行 JOIN 可读性差；两表分开语义清晰

### abogus 超时常量
- **D-02:** 在 `abogus_pool.go` 中**独立定义** `abogusPoolGetTimeout = 10 * time.Second`
  - 不复用 `sign_pool.go` 的 `signPoolGetTimeout`（避免跨文件依赖，两个池各自可调）
  - 值与 `signPoolGetTimeout` 相同（均 10s）
- **D-03:** `abogusPool.sign()` 改为镜像 `signPool.sign()` 的模式：
  ```go
  ctx, cancel := context.WithTimeout(context.Background(), abogusPoolGetTimeout)
  defer cancel()
  var entry *abogusEntry
  select {
  case entry = <-ap.pool:
  case <-ctx.Done():
      return "", fmt.Errorf("abogus pool: timed out waiting for available VM (pool may be exhausted)")
  }
  ```

### Benchmark 测试
- **D-04:** `BenchmarkGetStatsDetailed` 只测**新的单查询版本**，不保留旧的多次 QueryRow 版本作对比
  - 放在 `internal/db/db_test.go` 中
  - 使用内存数据库（`initMemoryDB` 或等价方式）预填若干条记录后跑 benchmark

### abogus 池耗尽测试
- **D-05:** 新增 `TestABogusPoolTimeout`，追加到现有 `internal/douyin/abogus_test.go`
  - 用 `resetABogusPool()` 重置全局池，创建 size=1 的测试池
  - 从池中取出唯一实例不归还（模拟池耗尽）
  - 调用 `sign()`，断言：在 10s+buffer 内返回包含 "timed out" 的错误
  - 思路与 `pornhub/client_test.go` 中 `TestGetModelVideos_ContextCancel` 一致

### Claude's Discretion
- Benchmark 的预填数据量（行数）
- 超时错误消息的具体文案（保持与 signPool 对称即可）
- `GetStatsDetailed` 的 `QueryRow` → `QueryRowContext` 是否升级（ctx 传入，非本 phase 要求）

</decisions>

## Specific Ideas

- `signPool.sign()` 是 abogus 改造的直接模板——两个函数完成后应该结构几乎对称
- `GetStats()` 现有的 `GROUP BY status` 方案（2 个查询：sources + downloads group by）可作参考，但 `GetStatsDetailed` 改为宽列聚合（不 GROUP BY，各状态用 CASE WHEN），更适合一次 Scan 到结构体字段
