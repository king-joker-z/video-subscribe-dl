## Correctness Analysis

### Counter Correctness

**bilibili:** `processOneJob` in `downloader.go` increments `platformCompleted["bilibili"]` when `result.Success` and `platformFailed["bilibili"]` when `result.Error != nil`. The `Platform` field is set to `"bilibili"` in all three `Job{}` constructions in `bscheduler`. This path is correct.

**douyin / pornhub:** These schedulers use their own download loops (not `Downloader.Submit`). The `scheduler.go` event-forwarding goroutines intercept `DownloadEvent` with `evt.Type == "completed"/"failed"` and call `s.dl.AddPlatformCompleted/Failed`. This is the only increment path for these two platforms and it is correct.

**No double-counting risk:** bilibili increments via `processOneJob` (inside `Downloader`); the event-forwarding goroutines for douyin/pornhub only handle events from their own `eventCh` channels, which are independent of `Downloader`'s internal event path. There is no overlap.

### Timestamp Correctness

`defer s.setLastCheckAt(time.Now())` at the top of `CheckSource` means the timestamp records _entry time_ of the check, not completion. This is consistent with D-06's stated semantics ("调度尝试时间"). Since `defer` runs after the function returns, `time.Now()` captures the time the function returns, not when it was called — this is slightly different from "entry time" but is practically equivalent for monitoring purposes.

**Zero-value safety:** The `IsZero()` guard in `HandlePrometheus` correctly prevents `time.Time{}.Unix()` (which is `-62135596800`) from being emitted. This was a required fix per D-10 and is correctly implemented.

### Thread Safety

- `lastCheckAt` on all three sub-schedulers uses a dedicated `sync.RWMutex` (`lastCheckMu`), consistent with the `pausedMu` pattern elsewhere in the same files.
- `platformCompleted/Failed` use `atomic.AddInt64` / `atomic.LoadInt64` without any mutex. This is correct since the underlying `*int64` pointers are write-once at construction time, and the map itself is never mutated after `New()`.
- `AddPlatformCompleted/Failed` have nil guards and access the map without a lock. The map is initialized once in `New()` and is never modified afterwards — safe for concurrent reads.
- `getSchedulerLastCheck` in `MetricsHandler` and `web.Server` are plain maps written only during startup (before any requests arrive) via setter calls in `setupRoutes()`. No concurrent write after server start. Safe.

### Test Coverage

`metrics_test.go` covers:

1. **`TestHandlePrometheus_PlatformCounters`** — Exercises `AddPlatformCompleted("bilibili")`, verifies counter appears in output with correct label syntax, verifies all three platforms are present (douyin/pornhub at 0), verifies HELP/TYPE lines for all new metric families, and verifies existing flat counters are preserved.
2. **`TestHandlePrometheus_SchedulerLastCheck_Zero`** — Verifies zero `time.Time{}` emits `0`, not `-62135596800`.
3. **`TestHandlePrometheus_SchedulerLastCheck_NonZero`** — Verifies a known Unix timestamp (`1700000000`) is rendered correctly.
4. **`TestHandlePrometheus_MethodNotAllowed`** — Verifies POST returns 405.

Coverage is sufficient for the new code paths. The `HandleMetrics` (JSON endpoint) is not tested, but it existed before this phase and its new paths (cooldown map) were already implicitly tested by the existing integration.

## Minor Observations (Non-blocking)

### O-1: `defer setLastCheckAt` captures _return_ time, not _entry_ time

```go
func (s *BiliScheduler) CheckSource(src db.Source) {
    defer s.setLastCheckAt(time.Now())   // time.Now() evaluated at deferred call, i.e. on return
    ...
}
```

`defer` with `time.Now()` as argument captures the value at the point where `defer` is registered (function entry), NOT at the time of execution. This is a Go language subtlety: `defer f(arg)` evaluates `arg` immediately when the `defer` statement is executed. So `time.Now()` is captured at function entry, not at return. This means the timestamp represents "when the check started", which is arguably better semantics than "when it finished" for an `event_last_check` gauge. The behavior matches D-06's stated intent. No bug — just worth noting for documentation clarity.

### O-2: Prometheus label quoting uses `%q`

```go
fmt.Fprintf(w, "vsd_downloads_completed_total{platform=%q} %d\n", p, dlStats.PlatformCompleted[p])
```

`%q` produces Go-style double-quoted strings (`"bilibili"`), which is valid Prometheus exposition format. The test correctly verifies `platform="bilibili"` (with double quotes). This is consistent with the existing `HandlePrometheus` style and Prometheus text format spec. ✅

### O-3: `HandleMetrics` JSON endpoint has no `lastCheckAt` fields

The JSON `/api/metrics` endpoint (not Prometheus) does not expose `lastCheckAt` or per-platform counts. This is within scope — the CONTEXT.md only required Prometheus format metrics. The JSON endpoint's `downloader` key will include `PlatformCompleted`/`PlatformFailed` via `Stats()`, which is a welcome bonus. No action needed.

### O-4: `getSchedulerLastCheck` map in `MetricsHandler` is not concurrency-safe for late registration

`SetSchedulerLastCheckFunc` writes to `h.getSchedulerLastCheck` without a mutex. This is safe only because all setters are called before `server.Start()` (single-threaded wiring phase), after which no further writes occur. The same pattern exists for `getCooldownInfo` / `getPHCooldownInfo`. This is an existing pattern accepted by the codebase — no action needed for this phase.

### O-5: `newTestDownloader` passes `(*bilibili.Client)(nil)` as the client

The test creates a `Downloader` with a nil Bilibili client. The worker goroutine launched by `New()` will immediately try to read from the queue channel, which is safe. The test only calls `AddPlatformCompleted` which doesn't touch the client. This pattern works because `Stop()` is registered as a `t.Cleanup` func. ✅
