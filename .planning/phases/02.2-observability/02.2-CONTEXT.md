# Phase 2.2: Observability - Context

**Gathered:** 2026-04-03
**Updated:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

在现有 Prometheus 端点（`GET /api/metrics/prometheus`）中新增三组指标：

1. **`vsd_downloads_completed_total{platform="..."}`** — 各平台下载成功计数（counter）
2. **`vsd_downloads_failed_total{platform="..."}`** — 各平台下载失败计数（counter）
3. **`vsd_scheduler_last_check_timestamp{platform="..."}`** — 各平台调度器最近一次扫描的 Unix 时间戳（gauge）

platform label 取值：`bilibili`、`douyin`、`pornhub`（与 DB `sources.type` 字段完全一致）。

不包含：单次下载进度指标、active/queued per-platform 拆分、通知/告警规则、指标持久化。

</domain>

<decisions>
## Implementation Decisions

### Platform 标签 — 下载计数器

- **D-01:** 在 `Job` 结构体中新增 `Platform string` 字段（取值 `bilibili`/`douyin`/`pornhub`）。
  各调度器在构造 `Job` 时从 `source.Type` 直接赋值。
  注意：ROADMAP.md 提及 "source_type field already carried by Job"，但实际代码中该字段**不存在**，必须新增。
  
- **D-02:** 在 `Downloader` 结构体中新增 `platformCompleted map[string]*int64` 和 `platformFailed map[string]*int64`，
  在 `New()` 中用已知 platform 列表初始化（3 个 key，每个指向新分配的 int64）。
  `processOneJob` 完成/失败时按 `job.Platform` 做原子 +1，需加 nil 守卫（见 D-11）。
  与现有 `totalCompleted`/`totalFailed` 模式对称，不替换现有平坦计数器（保持向后兼容）。

- **D-03:** `DownloaderStats.Stats()` 返回值扩展，暴露 per-platform 计数，供 `HandlePrometheus` 读取。

- **D-11:** 在 `Downloader` 上新增两个公共方法：
  `AddPlatformCompleted(platform string)` / `AddPlatformFailed(platform string)`。
  这是 douyin 和 pornhub 计数器的唯一递增入口（因为这两个调度器完全不使用 `downloader.Submit()`）。
  bscheduler 通过 `processOneJob` 自动递增（Job.Platform = "bilibili"）；
  dscheduler 和 phscheduler 由 `scheduler.go` 的事件转发 goroutine 递增（见 D-12）。

- **D-12:** 在 `internal/scheduler/scheduler.go` 的 douyin 事件转发 goroutine 中，
  当 `evt.Type == "completed"` 时调用 `s.dl.AddPlatformCompleted("douyin")`，
  `evt.Type == "failed"` 时调用 `s.dl.AddPlatformFailed("douyin")`。
  ph 事件转发 goroutine 同理，使用 `"pornhub"`。
  这保持了计数逻辑集中，无需给 dscheduler/phscheduler 注入 Downloader 引用。

### 调度器 Last-Check Wiring

- **D-04:** 每个子调度器（`phscheduler`、`bscheduler`、`dscheduler`）新增 `lastCheckAt time.Time` 字段
  和 `lastCheckMu sync.RWMutex`（专用 mutex，不复用其他 mutex）、
  `LastCheckAt() time.Time` 公共方法和 `setLastCheckAt(t time.Time)` 私有方法。

- **D-05:** `MetricsHandler` 新增 `SetSchedulerLastCheckFunc(platform string, fn func() time.Time)` 方法，
  镜像现有 `SetCooldownInfoFunc` / `SetPHCooldownInfoFunc` 模式。
  在 `router.go` 的 `SetSchedulerLastCheckFunc` 代理方法和 `web/server.go` 的 `setupRoutes()` 阶段
  为三个平台分别注册（见 D-13）。

- **D-06:** `lastCheckAt` 在 `CheckSource()` 的 **顶部** 用 `defer s.setLastCheckAt(time.Now())` 更新，
  对三个调度器统一适用。
  语义为"最近一次 *调度尝试* 时间"——IsPaused、冷却期等提前返回也会更新时间戳，
  这对运维可见性是可接受的（反映"调度器活跃"而非"实际完成扫描"）。
  首次运行前返回零值 `time.Time{}`；见 D-10 关于零值的输出处理。

- **D-13:** Wiring 路径新增 `web/server.go` 层（原 CONTEXT.md 遗漏此文件）：
  - `web/server.go`：新增 `getSchedulerLastCheck map[string]func() time.Time` 字段；
    在 `NewServer()` 中初始化该 map；新增 `SetSchedulerLastCheckFunc(platform string, fn func() time.Time)` 方法；
    在 `setupRoutes()` 中遍历 map 调用 `s.apiRouter.SetSchedulerLastCheckFunc(platform, fn)`。
  - `cmd/server/main.go`：调用 `server.SetSchedulerLastCheckFunc("bilibili", sched.GetBiliLastCheckAt)` 等三处。
  - `internal/scheduler/scheduler.go`：新增 `GetBiliLastCheckAt()`, `GetDouyinLastCheckAt()`, `GetPHLastCheckAt()` 代理方法。

### Metric 命名与 Label 约定

- **D-07:** 严格按 ROADMAP.md 指定的 metric 名称实现：
  - `vsd_downloads_completed_total{platform="..."}` — counter
  - `vsd_downloads_failed_total{platform="..."}` — counter
  - `vsd_scheduler_last_check_timestamp{platform="..."}` — gauge（Unix 秒）
  
- **D-08:** Platform label 值与 DB `sources.type` 完全一致：`bilibili`、`douyin`、`pornhub`。
  无缩写映射，无翻译层。平台循环输出顺序：字母序（bilibili → douyin → pornhub）。

- **D-09:** 现有平坦计数器 `vsd_downloader_completed_total` 和 `vsd_downloader_failed_total`
  保持不变，不重命名，避免破坏已有 scrape 配置。

### 零值处理（Critical）

- **D-10:** `time.Time{}.Unix()` 返回 `-62135596800`（公元1年的 Unix 时间戳），**不是** `0`。
  `HandlePrometheus` 必须用 `IsZero()` 检查，而非直接调用 `.Unix()`：
  ```go
  ts := int64(0)
  if fn := h.getSchedulerLastCheck[p]; fn != nil {
      if t := fn(); !t.IsZero() {
          ts = t.Unix()
      }
  }
  fmt.Fprintf(w, "vsd_scheduler_last_check_timestamp{platform=%q} %d\n", p, ts)
  ```
  同样，当 `fn == nil`（setter 未被调用）时输出 `0`。

### 计数器持久化

- **D-14:** 计数器仅保存在内存（原子 int64），服务重启后归零。
  与现有行为一致；Prometheus 使用 `rate()` / `increase()` 函数可自动处理 counter reset。
  不写 SQLite，不增加 DB I/O。

### nil 守卫（Critical）

- **D-11-guard:** `processOneJob` 中递增 per-platform 计数器时必须双重守卫：
  ```go
  if p := job.Platform; p != "" {
      if ptr := d.platformCompleted[p]; ptr != nil {
          atomic.AddInt64(ptr, 1)
      }
  }
  ```
  防止老代码路径中 `job.Platform == ""`（map 取 nil 指针）导致 panic。
  bscheduler 有 **3 处** `downloader.Job` 构造（`process.go` 2 处 + `retry.go` 1 处），
  均需补充 `Platform: "bilibili"`。

### Claude's Discretion

- `handlePrometheus` 中平台循环的枚举顺序（`bilibili`、`douyin`、`pornhub` 字母序即可）
- HELP / TYPE 注释的具体 comment 文案（与现有 Prometheus 块风格保持一致）
- 测试中 mock 调度器 LastCheckAt 的具体实现方式
- `lastCheckMu` 命名（只需与同文件其他 mutex 保持一致风格）

<code_context>
## Existing Code Insights

### Reusable Assets
- `SetCooldownInfoFunc` / `SetPHCooldownInfoFunc`：MetricsHandler 现有 setter 注入模式，`SetSchedulerLastCheckFunc` 直接仿照
- `totalCompleted` / `totalFailed` int64 + `atomic.AddInt64`：Downloader 现有计数器原子操作模式，per-platform 计数器照此模式扩展
- `Source.Type` 字段（`"bilibili"` / `"douyin"` / `"pornhub"`）：可直接赋值给 `Job.Platform`，无需映射
- `pausedMu sync.RWMutex`：各调度器现有 RWMutex 模式，`lastCheckMu` 照此添加

### Established Patterns
- Prometheus 输出格式：`# HELP ... \n# TYPE ... \n<metric> <value>\n`，`HandlePrometheus` 现有实现风格需保持一致
- Counter 用 `atomic.AddInt64`；gauge 用 `atomic.LoadInt64` 或加 mutex
- 跨组件回调注入：`SetXxxFunc(fn func() ...)` setter 已是本项目标准模式
- Server 层持有 callback map + `setupRoutes()` 注入：`web/server.go` 现有模式

### Integration Points
- `processOneJob` 中 `atomic.AddInt64(&d.totalCompleted, 1)` 和 `atomic.AddInt64(&d.totalFailed, 1)` 是 bilibili per-platform 计数的插入点
- `scheduler.go` 中 douyin/ph 事件转发 goroutine（`evt.Type` switch）是 douyin/pornhub 计数的插入点
- 三个子调度器的 `CheckSource()` 顶部（`defer setLastCheckAt`）是 lastCheckAt 更新点
- `router.go` New() 函数 → `web/server.go` setupRoutes() → `cmd/server/main.go` 是 MetricsHandler setter 注册的完整 wiring 链
